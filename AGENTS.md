# Personal Career OS — Agent 工作规范

> **关于本规范**：这是一套开放标准，所有支持读取 `AGENTS.md` / `CLAUDE.md` 的 AI agent 都可以直接使用本项目。本规范定义了目录约定、交互路由、执行铁律和红线，agent 按规范操作即可完成求职材料的全流程生成。

个人求职操作系统：把用户的经历沉淀成结构化记忆库，针对任意 JD 产出「复制粘贴就能投」的全套材料（一页简历 DOCX/PDF/PNG + 投递邮件 + 链接），并把每次投递中新确认的事实回写记忆库。

本文件是所有 AI agent 在本仓库工作的统一规范。

## 目录约定

| 目录 | 内容 | 规则 |
|---|---|---|
| `profile/` | 个人经历库（唯一事实源） | 简历上每个字必须能溯源到这里的事实清单；新事实必须回写 |
| `profile/experiences/` | 一段经历一个文件 | frontmatter（类型/标签/适配方向/完整度）+ 事实清单 + 现成表述 + 待挖掘 |
| `methodology/` | 简历方法论 + 默认版式规范 + 风格库 + 岗位速查 + 挖掘问题库 | 所有改写、JD 分析、排版、邮件生成遵守它们 |
| `workflows/` | 各工作流的执行手册 | agent 按「交互路由」读对应文件并严格执行 |
| `template/` | 简历模板 = 当前标准版简历 | 只有一个 `resume.html`；用户未指定版式时固定使用其结构，改动后必须渲染验证 |
| `scripts/` | 导出与渲染脚本 | HTML → DOCX/PDF/PNG；三种格式分别做一页和视觉校验 |
| `applications/` | 投递记录，一次一个目录 | 命名 `YYYY-MM-DD-公司-岗位`；`_log.md` 是台账，只追加 |
| `archive/` | 原始材料 | 只进不改不删 |

## 交互路由（用户无需记命令）

用户直接说话，agent 按意图路由到对应工作流：

| 用户在做什么 | 执行哪个工作流 |
|---|---|
| 粘贴了一段 JD、发了职位链接、说想投某个岗位 | `workflows/apply.md`（投递全流程） |
| 提到新实习 / 新比赛 / 新证书 / 新链接，或发来简历、成绩单等材料 | `workflows/intake.md`（沉淀入库） |
| 说自己没有简历、没什么经历，或第一次用不知道从哪开始 | `workflows/intake.md` 的「零起点建库」：访谈式普查代替材料，产出第一份标准版简历 |
| 想补某段经历的细节，或某段经历太薄需要充实 | `workflows/dig.md`（深挖） |
| 要 Word/DOCX、PDF、PNG/图片或完整投递包 | `workflows/export.md`（多格式导出） |
| 问投递进展、要更新某次投递的状态 | 直接编辑 `applications/_log.md` |

意图不确定时（比如粘了 JD 但可能只是想讨论岗位），先用一句话确认再动手，不要猜错方向白做一轮。

**快捷命令**（可选）：支持自定义命令的 agent（如 Claude Code）可使用 `.claude/commands/` 下的快捷命令（`/apply`、`/intake`、`/dig`），实际执行仍以 `workflows/` 为准。

**交互方式适配**：
- 支持结构化选项的 agent（如 Claude Code 的 `AskUserQuestion`）：在关键决策点（选经历、确认方案等）用结构化选项让用户点选
- 不支持的 agent：列出选项编号或文字描述，让用户回复编号或关键词

## 铁律

1. **保真**：不编造经历、数字、头衔。数字缺失就问本人，没确认就不写。profile 中带【待确认】的事实，用之前必须先问清。
2. **一页 A4**：DOCX、PDF、PNG 都必须恰好一页；不能只验证 PDF：
   - Windows：`pwsh scripts/render.ps1 -Html <文件.html>`
   - macOS / Linux：`bash scripts/render.sh <文件.html>`
   - DOCX：用 `scripts/export-docx.py` 生成后，运行 `scripts/verify-docx.py` 做一页硬校验并查看 QA 图片；出现第 2 页即失败
   超页删内容并重新生成全部格式，不允许字号 < 10pt。PDF 与 PNG 必须同源；最终查看 PDF 同源 PNG 和 DOCX 渲染图检查排版。
3. **沉淀闭环**：会话中用户确认的任何新事实，结束前写回 `profile/`，并同步 `_index.md`。
4. **时效检查**：`profile/preferences.md` 里的到岗时间等信息有时效，引用前核对当前日期，过期先问。
5. **视觉素材由用户选择**：校徽和证件照都是选做，不能因为模板里有位置就自动添加。生成简历前必须给用户一次明确选择；选择结果记录到 `profile/basics.md`。证件照只能使用用户提供的文件；校徽使用用户提供的文件，或仅在用户明确要求搜索时进行 Web Search，展示候选来源并由用户确认后再使用。用户选择“不需要”时保持页眉对应位置为空，禁止放占位图、假图片或自行搜索。
6. **固定的是版式，不是图片**：用户未指定其他模板时，严格执行 `methodology/默认简历版式规范.md`。重点校验字体、字号、行距、页边距、区块间距、分割线、日期对齐和项目符号；不得为了填满页面擅自加入装饰元素。

## 命名

- 简历输出文件：默认 `学校-姓名-专业-年级`；JD 有指定命名一律按 JD
- 拿到 JD 后必须把命名模板中的字段替换成用户真实信息，得到一个**最终文件名**（例如 `李华-XX大学-金融学.pdf`）。只提取「姓名-学校-专业」模板、不落实为实际文件名，视为未完成
- 最终文件名必须从 JD 分析一路传递到渲染命令、`email.md` 附件清单和最终汇报；投递目录下调用渲染脚本时禁止省略 `-Name` / 输出基名参数
- DOCX、PDF、PNG 使用同一输出基名；交付前必须检查磁盘上的实际文件名与最终文件名逐字一致；不一致时不得汇报「已完成」
- 经历文件名：中文短名（如 `XX证券-行研实习.md`）；代码、脚本、命令名用英文

## 红线（必须先问用户）

- 删除 profile/ 或 applications/ 下的任何文件
- git push、对外发布任何材料
- 把简历/邮件实际发送出去（本项目只准备材料，发送永远由用户手动完成）
