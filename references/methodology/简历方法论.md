# 简历方法论

> 本项目所有简历改写、JD 匹配、邮件生成都遵守本文件。配套文件：《[风格库](风格库.md)》（企业性质写法）、《[岗位速查](岗位速查.md)》（岗位专业词与专项入口）、《[挖掘问题库](挖掘问题库.md)》（提问清单）。
> 来源说明：方法论框架受 [industry-resume-toolkit](https://github.com/shangsitongshizaitiantang/industry-resume-toolkit)（CC BY-NC-ND 4.0）与 [Resume-Tailor-AI](https://github.com/JaimeYeung/Resume-Tailor-AI) 启发，全部内容为本项目原创提炼表述，未复制原文。

## 三条铁律

1. **保真**：简历上每个字都要经得起面试追问。判断标准只有一个——被追问三层还不露馅才能写。数字必须来自 profile/ 事实清单；缺数据就问本人，本人没确认就不写，绝不推测补数。职责可以重新组织语言，不能升格（实习生 ≠ 负责人）。
2. **一页 A4**：硬约束。DOCX、PDF、PNG 都必须恰好一页，不能只验证 PDF。PDF/PNG 用渲染脚本校验，DOCX 另行渲染为逐页图片校验；任一格式超页就删减内容并全部重导出，不允许用缩小字号（< 10pt）或压行距蒙混。
3. **价值翻译**：简历是写给 HR 和用人经理看的，不是工作日志。每条经历回答的问题是「我为什么对你有用」，不是「我干过什么」。

## 写作坐标：岗位 × 行业 × 公司性质 × JD

同一段事实不能只换几个关键词就投所有岗位。动笔前先确定四件事：

1. **岗位**决定要证明什么能力。行研看研究闭环，产品看用户与落地，数据分析看数据规模与业务结论。
2. **行业**决定研究对象、数据口径、关键指标和专业词。新能源写产业链、招标、价格与产能；消费写品类、渠道、客群与复购；互联网写用户、场景、转化与留存。
3. **公司性质**决定证据排序和语气。券商看严谨口径与研究产出，大厂看规模与业务影响，国央企看完整稳重，外企看清晰结构与结果。
4. **JD**决定本次材料的关键词、经历顺序和取舍，但不能反过来创造用户没做过的事。

参考项目采用“通用规则 + 企业性质 pattern + 岗位 pattern”的叠加方式。本项目沿用这个骨架，再把目标行业作为岗位规则内的业务语境：**真实事实 → 通用价值翻译 → 公司整体调性 → 岗位证明结构 → 行业对象与指标 → JD 语言**。优先级固定为：**保真 > 核心写作方法 > 岗位能力 > 行业语境 > 公司性质 > JD 关键词 > 版面**。具体规则可以细化核心方法，不能推翻保真和价值翻译。

## 改写三步

### 1. 排雷（信息卫生）

- 模糊量词清零：「很多、大量、丰富、多次、诸多」→ 换成具体数字，拿不到数字就换写法
- AI 味虚词清零：「赋能、打造、深耕、多维度、无缝衔接、抓手」不出现
- 每条以动词开头，不写「我」；语气克制，形容词能删就删
- 备考中的证书不写；已过部分科目写「XX（已通过 N 科）」
- Office 不拆成 Word/Excel/PPT 单列；真正的差异化工具（Wind、iFinD、Python）单独写
- 成绩要么成体系放，要么不放，不挑高分放
- 联系方式一行解决，不放二维码、家庭住址、多个邮箱

### 2. 价值翻译

- 公式：**动作 + 落地方式 + 量化结果**。三要素缺一就是半成品 bullet
- 动作 → 业务意义：「整理数据」→「维护 XX 数据库，支撑 X 期复盘汇报」
- 量化维度任选其一：数量 / 频率 / 规模 / 结果 / 名次 / 覆盖面
- 关键词对齐：用 JD 里反复出现的动词和名词改写表述（对齐 ≠ 堆砌，一条 bullet 最多命中 1-2 个关键词）
- 关联度排序：与 JD 强相关的经历放前、写满；弱相关的压成一行或删除
- 行业落地：不能停在「完成研究、整理资料、支持分析」；写清研究对象、数据来源/方法、判断或产出，以及它服务了什么问题
- 公司适配：不是更换形容词，而是重新排列证据。目标公司最看重的证据放在首条，弱证据压缩或删除

### 3. 深挖（信息不够时向本人提问）

六个维度，一轮最多问 3-5 个，只问事实、不问动机和感受：

1. 业务背景 —— 这是什么业务？服务谁？
2. 规模数据 —— 处理了多少条 / 多少篇 / 多少人 / 多少钱？
3. 个人贡献 —— 你（不是团队）具体做了什么？
4. 成果 —— 结果如何？有没有数字或者反馈？
5. 困难与解决 —— 卡在哪里？怎么解决的？
6. 关键判断 —— 你自己做过什么决定？

答案一律回写进 `profile/experiences/` 对应文件的事实清单。按经历类型的具体追问角度和「全面普查」清单见《[挖掘问题库](挖掘问题库.md)》。

## JD 分析框架

拿到 JD 先提取九类信息（软技能词如「沟通、团队合作」不算关键词，不提取）：

| 类别 | 提取内容 |
|---|---|
| 硬性门槛 | 学历 / 专业 / 年级 / 证书 / 每周天数与持续时长 / 必备工具 —— 逐条比对，不满足要明示风险 |
| **文件命名要求** | **JD 中明确要求的简历文件命名格式**（如「姓名-学校-专业.pdf」「学校+姓名+应聘岗位」等）—— 没有要求时默认 `学校-姓名-专业-年级` |
| 职能关键词 | 岗位名及其变体，判断这个岗位属于哪个职能族 |
| 目标行业 | 判断业务所处行业、产业链位置、研究对象和核心指标；不要只记录公司名称 |
| 公司性质与读者 | 券商/买方/银行/互联网/国央企/外企/创业公司；材料最终给 HR、研究员、业务负责人还是产品负责人看 |
| 硬技能 | 工具、软件、方法名 —— 能被机器筛精确匹配的词 |
| 动作动词 | 职责描述里反复出现的动词（跟踪 / 搭建 / 输出 / 落地…） |
| 领域术语 | 行业黑话与业务语境词（产业链环节、指标名、品类名） |
| 加分项与风格信号 | 标「优先 / 加分」的条目；企业性质与文化用语 → 对应《风格库》选节 |

**命名要求常见表述**：「请以"姓名-学校"命名」「简历命名格式：学校+专业+姓名」「附件命名：姓名_学校_岗位」「文件名请注明应聘岗位」等。提取后必须立刻把字段替换为真实信息，形成包含扩展名的最终文件名；不能把未实例化的格式模板当成已完成。模板缺值或口径不明时先问用户。第一次遇到某公司的命名要求时，记入《岗位速查》对应公司章节。

按出现频次和位置（标题 > 职责首条 > 任职要求）排出 **Top 10 关键词**，再把本人经历对着 Top 10 分三类（区分后两者是关键）：

- **已覆盖**：简历直接能证明的
- **可挖**：做过但没写出来 / 表达不到位的 —— 靠深挖和改写解决
- **真缺失**：没做过的 —— 保真铁律，不编，考虑用相邻能力弱化处理

## 岗位、行业与公司性质的组合

先执行本文件的通用改写，再叠加《[风格库](风格库.md)》中的公司性质和《[岗位速查](岗位速查.md)》中的岗位规则；岗位专项文件内部再按目标行业调整对象、口径和指标。不能只套“大厂风”“券商风”，也不能只替换一组专业词。

行研尤其要先判断研究成果给谁用。具体边界见《[行业研究 / 投资研究专项](岗位/行业研究.md)》；该文件在岗位内部区分不同企业场景，再叠加《风格库》的整体语气。

## 一页纸内容预算（默认模板下）

| 区块 | 预算 |
|---|---|
| 页眉（姓名+联系） | 2 行 |
| 教育背景 | 3-4 行 |
| 实习经历 | 每段 3-4 条 bullet，每条 ≤ 2 行 |
| 实践与竞赛 | 3-4 个条目，每个 1-2 行 bullet |
| 技能与证书 | 3-4 行 |
| 补充信息（可选） | 1 行（到岗时间等） |

超页删减顺序：弱相关竞赛 → 校园工作 → 爱好 → 压缩 bullet 第二行 → 与本人商量。

## 零实习简历（素材薄时的内容策略）

没有实习 ≠ 没有简历。HR 对低年级/零实习候选人看的是能力证据，不是头衔：

- **区块按素材改名，不留空区块**：没有实习就没有「实习经历」区，用「项目经历」「实践与竞赛」「校园与志愿」等组合；禁止写「暂无」、禁止硬凑
- **课程项目按实习规格写**：抬头 = 项目名 + 角色（课程名放辅助位置），bullet 照套「动作 + 落地方式 + 量化」——几人组、数据量、交付物、下落（展示/获奖/被采纳）
- **教育背景可升格**：与目标方向相关的 3-5 门课 + 成体系的成绩 + 毕设/论文方向，写满预算没问题
- **个人实践有记录就是正经经历**：持续的投资/交易记录、自媒体数据、自学做出的可展示交付物，按实践区块写，量化维度照常
- **一页是上限，不是 KPI**：素材只够 2/3 页就 2/3 页；不用「自我评价」凑版面，不放水词
- **普查完仍对不上目标方向**：如实告知缺口，给补素材路径（比赛/证书/可展示的自学项目）；没发生的事不进简历

## 投递邮件规范

- **主题**：JD 指定格式优先；默认 `应聘【岗位】-学校-姓名-每周X天-可实习X个月`
- **正文三段**，总长 ≤ 200 字：
  1. 我是谁 + 投什么岗（1-2 句）
  2. 三个匹配点（对齐 JD 硬性要求和关键词，每点一句，可用短横线列表）
  3. 到岗时间 + 附件说明 + 礼貌收尾
- **署名**：姓名 / 学校 / 电话（见 profile/preferences.md）
- **附件命名**：默认 `学校-姓名-专业-年级.pdf`；JD 有指定格式一律按 JD。邮件附件清单、磁盘文件名和最终汇报必须引用同一个已实例化的最终文件名
- 有匹配的作品集 / GitHub 链接（profile/links.md）时放正文第二段末尾
- `email.md` 末尾另给一段 **平台私信/微信简版**：不写邮件格式，控制在 100 字以内，保留应聘岗位、最强匹配点、到岗信息和简历附件提示
- 正文里提到的附件格式必须与实际生成文件一致；默认正式附件用 PDF，只有对方明确要求可编辑版本时才优先提 DOCX
