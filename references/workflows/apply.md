# 投递工作流（research / apply）

> research：用户只要求研究、看看或分析 JD 时，仅执行第 0-3 步并给出结论，不创建或修改文件。
> apply：用户明确要求准备投递、定制或生成材料时，执行完整流程。
> 目标：先给出可核验的岗位判断；确认进入 apply 后，再产出「复制粘贴就能投」的全套材料。

路径统一按根目录 `SKILL.md` 的 `SKILL_ROOT` / `CAREER_ROOT` 约定解析。

## 0. 拿到 JD

- 用户消息里有 JD 文本 → 直接用；给的是文件路径 → 读文件
- 都没有 → 请用户把 JD 完整粘贴过来（公司名、岗位名一起）

## 1. 读上下文（顺序固定）

1. `references/methodology/简历方法论.md` —— 全程遵守
2. `profile/_index.md` → `basics.md`、`preferences.md`、`links.md`
3. 按 JD 方向挑出相关经历，读对应 `profile/experiences/*.md`
4. 按目标企业性质读 `references/methodology/风格库.md` 对应节；按岗位读 `references/methodology/岗位速查.md` 对应节

注意 `preferences.md` 中带时效的信息（到岗时间等），过期就先问用户更新。

**空库分流**：如果 `profile/_index.md` 的经历表只有占位行，且 `profile/experiences/` 下除 `_TEMPLATE.md` 外没有经历文件：

- research 模式仍继续做岗位门槛、关键词、工作内容和通用风险分析；个人匹配项统一标记为“资料不足”，结尾给出“导入简历 / 从零建档”两个入口。
- apply 模式才暂停生成，带用户走 `references/workflows/intake.md`。禁止拿 `template/resume.html` 的示例内容投递。
- 用户同一条消息同时给了简历和 JD 时，连续完成“导入建档 → JD 分析 → 定制”，不要让用户重复发送材料。

## 2. JD 分析（展示给用户）

按方法论的 JD 分析框架输出：硬性门槛比对（逐条 ✓/✗）、高频关键词、加分项、风格信号、**文件命名要求**（如有）。

**文件命名要求提取**：
- JD 中明确要求的命名格式（如「姓名-学校-专业.pdf」「学校-姓名-岗位」等）
- 没有要求时，默认用 `城市-学校-姓名-专业-年级`
- 第一次遇到某公司的命名要求时，记入本次 `notes.md`；不要修改 Skill 自带的方法论

硬性门槛有 ✗ 时明确指出风险，让用户决定是否继续。

## 3. 经历选配（和用户确认后再动笔）

给出方案并等用户确认：

- 选哪几段经历、排什么顺序、每段主打什么角度（引用经历文件里的「现成表述」角度）
- 舍弃/压缩哪些、为什么
- 若关键经历「完整度：低/中」且 JD 高度相关：现场按方法论「深挖六问」问 3-5 个事实问题（答案回写经历文件），或建议先走深挖工作流
- 涉及【待确认】标记的事实（如证书表述、专业口径）第一次用到时必须问清，确认后更新 profile

**交互方式**：
- 支持结构化选择的 agent：用当前界面能力让用户点选
- 不支持的 agent：列出选项编号，让用户回复编号或文字

用户确认后才进入生成。用户如说「直接出」，跳过讨论但【待确认】事实仍必须问。

## 4. 生成简历内容源

1. 建目录 `applications/YYYY-MM-DD-公司-岗位/`（当天日期）
2. 复制 `template/resume.html` 进去，只改内容不改版式骨架；该 HTML 是本次 DOCX/PDF/PNG 的共同定稿内容源
3. 改写规则：方法论三步（排雷 → 价值翻译 → 量化），关键词对齐 JD，风格对齐企业性质
4. **保真**：每个数字都能在经历文件事实清单里找到；缺的问用户，没确认就不写
5. JD 有硬性时间要求时，在技能区下加一行「补充信息」（到岗时间、每周天数，从 preferences.md 取）
6. links.md 中「适用场景」匹配的链接写进简历对应位置

## 5. 生成多格式简历并校验

读取并执行 `references/workflows/export.md` 的版本定位、格式生成和校验部分，在同一个 `applications/YYYY-MM-DD-公司-岗位/` 投递包目录生成：

- `resume.html`：共同定稿内容源；
- `<简历文件名>.docx`：可编辑 Word 版；
- `<简历文件名>.pdf`：正式附件；
- `<简历文件名>.png`：手机端/微信图片。

**文件名规则**（优先级从高到低）：
1. **JD 明确要求的格式优先**（如「姓名-学校-专业」「学校+姓名+应聘岗位」等）—— 从第 2 步提取的命名要求
2. 没有要求时，默认 `城市-学校-姓名-专业-年级`（从 preferences/basics 取）
3. 命名中的空格和特殊字符统一替换为 `-`

- PDF/PNG 与 DOCX 都要做一页和视觉校验。超页时按方法论删减顺序删内容重导出，禁止把字号降到 10pt 以下。

## 6. 生成配套材料（同目录）

- `jd.md`：JD 原文 + 第 2 步的分析
- `email.md`：按方法论写好 **邮件主题**、**邮件正文**、一段更短的**平台私信/微信投递正文**，末尾列附件清单
- `notes.md`：用了哪些经历/角度、关键词覆盖情况、本次讨论中的遗留问题

## 7. 记账与沉淀（不能跳过）

1. 在 `applications/_log.md` 追加一行（日期/公司/岗位/渠道/状态=已准备）
2. **回写**：本次对话中用户确认的新事实 → 写进对应经历文件的事实清单，消掉已回答的「待挖掘」；涉及完整度变化就更新 `_index.md`
3. 全新经历 → 提示用户之后走沉淀工作流补录

## 8. 汇报

最终消息里给出：产物文件清单（可点击路径）、3 个关键定制点、复制粘贴指引（邮件主题贴哪、正文贴哪、Word/PDF/图片分别用于什么）。把 `applications/YYYY-MM-DD-公司-岗位/` 称为“投递包目录”，不要称为新的 Git 仓库。
