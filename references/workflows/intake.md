# 沉淀工作流（intake）

> 触发：用户提到新实习/比赛/证书/链接，或发来简历、成绩单等材料；也包括首次使用但什么材料都没有（零起点建库）。
> 目标：任何关于「我」的新信息，进来一次就永久沉淀，之后所有投递自动可用。

路径统一按根目录 `SKILL.md` 的 `SKILL_ROOT` / `CAREER_ROOT` 约定解析。

## 1. 拿材料

- 用户给了文件路径（docx/pdf/md/txt）→ 优先用当前 Agent 的文档/PDF 读取能力完整提取正文、表格和链接；没有专用能力时，DOCX 可用 Python `zipfile` 解 `word/document.xml` 兜底
- 是口述文字 → 直接用
- 都没有：
  - 符合下方「空库判定」→ 走「1b. 零起点建库」，用访谈代替材料
  - 库不空 → 问用户：「这次要沉淀什么？发文件或直接说都行。」

**空库判定**：`profile/_index.md` 的经历表只有占位行，且 `profile/experiences/` 下除 `_TEMPLATE.md` 外没有经历文件。不要把占位文案当成真实经历。

## 1b. 零起点建库（没有简历、没有任何材料）

没有简历不是障碍：素材靠问出来，不靠用户交上来。「哪些事算经历」由 agent 判断，不要让用户自己预判。

1. **开场定预期**（一句话）：「不用先有简历。我每轮只问 3-5 个事实问题；随时可以暂停，答过的都会保存，下次接着来。」
2. **立基座**：第一轮只问姓名或希望使用的称呼、学校/专业/年级及起止时间；姓名可以暂缓，电话和邮箱推迟到首次生成简历或投递材料时再补。再问求职方向、目标城市、可实习时间 → 分别落 `profile/basics.md` / `preferences.md`。方向说不清就记「未定」，普查完按素材反推适配方向给建议。
3. **六轮普查**：按 `references/methodology/挖掘问题库.md` B 节逐轮问，一轮 3-5 问，答完一轮落一轮。用户答「没有」时用 B 节的降级问法换更具体的场景再试一次；连问两次确实没有就跳过，不逼问。
4. **可中断续跑**：用户中途喊停 → 已答内容照常按第 4-5 步落库，在 `profile/_index.md` 待办记「零起点普查进行到第 N 轮」，下次从那继续。
5. 普查走完 → 按 4-5 步落库，然后**必须执行第 6 步**——零起点用户的核心交付物就是第一份标准版简历。

## 2. 与现有 profile 做 diff

（零起点建库跳过本步和第 3 步：空库没有可 diff 的，普查已代替补缺口。）

读 `profile/_index.md` 和相关文件，把新材料内容分三类：

- **已有**：跳过
- **冲突**：新旧说法不一致（数字、头衔、时间）→ 列出来问用户以哪个为准，答案落库并注明来源
- **新增**：新经历或新事实

## 3. 补缺口（一轮 3-5 问）

对新增内容按方法论「深挖六问」问最关键的 3-5 个问题；如果是首次建库或材料很薄，用 `references/methodology/挖掘问题库.md` B 节的六轮普查清单。只问事实（做了什么、多少、结果），不问动机和感受。用户不想答就先落库、缺口记入「待挖掘」。

## 4. 落库

- 新经历 → 在 `profile/experiences/` 建文件，**参考 `_TEMPLATE.md` 的格式**：frontmatter（类型/组织/角色/时间/标签/适配方向/完整度）+ 一句话 + 事实清单 + 现成表述 + 待挖掘
- 已有经历的新事实 → 追加进事实清单，标注来源；能升级「现成表述」就顺手打磨
- 基础信息类（证书、技能、链接）→ 更新 `basics.md` / `links.md` / `preferences.md`
- 链接类信息一律登记进 `links.md` 并写清「适用场景」

## 5. 更新索引

`profile/_index.md`：新经历加行、完整度调整、全局待办增删。

## 6. 首次建库收尾：生成标准版简历

只要 `template/resume.html` 还是示例内容（「张三」），建完库就必须做本步——库建好了，用户就该有一份能直接用的标准版简历：

1. 先给用户一次页眉视觉选择：证件照可选上传并使用/不使用，且只能用用户提供文件；学校校徽可选上传并使用/帮我搜索/不使用，只有明确选择搜索时才能展示候选并等待确认。两项互相独立，均可不使用；将选择和来源记录在基础信息中。
2. 用 profile 的真实内容替换 `template/resume.html` 的全部示例内容，只改内容和用户确认使用的页眉图片，不改版式骨架
3. 如果姓名、电话或邮箱仍未填写，说明它们只用于简历页眉并一次性补问；用户明确不要展示的字段就删除，不要自行填占位符
4. 写法遵守 `references/methodology/简历方法论.md`（排雷 → 价值翻译 → 量化）；没有 JD 时按 `preferences.md` 的主线方向组织每段经历的角度
5. 没有实习就不留空的「实习经历」区块：区块标题按素材改名/重组（如「项目经历」「实践与竞赛」「校园与志愿」），细则见方法论「零实习简历」节
6. 读取并执行 `references/workflows/export.md`，默认生成标准版 HTML、DOCX、PDF、PNG。分别查看 HTML/PNG 和 Word 渲染结果；超页删内容，素材不满一页也不注水——**一页是上限，不是指标**
7. 提醒一次隐私：profile 和模板里已是真实信息，仓库不要公开推送（见 README 隐私提醒）

## 7. 汇报

- 用用户语言说明已沉淀了哪些经历、技能或求职偏好，以及还有哪些事实值得之后深挖；不要报告内部文件名或路径
- 首次建库：用“可编辑 Word 简历”“正式 PDF 简历”“手机端图片”等可读标签交付成品，并说明各自用途；只有用户明确要求时才展示实际文件名或路径
- 零起点用户追加「素材缺口」：普查后目标方向仍没有强素材就如实说，给 1-3 条补素材建议（能报名的比赛、能考的证书、能做出可展示交付物的自学项目）。建议只是行动参考——**没发生的事一个字不进简历**。
