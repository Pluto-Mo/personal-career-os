# 投递工作流（apply）

> 触发：用户粘贴 JD、发职位链接、或说要投某个岗位。
> 目标：产出「复制粘贴就能投」的全套材料。

## 0. 拿到 JD

- 用户消息里有 JD 文本 → 直接用；给的是文件路径 → 读文件
- 用户上传的是 PDF、Word 或图片 → 读取完整文件，不只摘岗位职责；命名要求常出现在投递方式、页脚、图片底部或邮件说明中，这些区域也必须检查
- 附件存在无法读取、截断或图片文字模糊的情况 → 明确告诉用户并请其补充原文；未完整检查附件前不得判断「JD 未指定命名」
- 都没有 → 请用户把 JD 完整粘贴过来（公司名、岗位名一起）

## 1. 读上下文（顺序固定）

1. `methodology/简历方法论.md` —— 全程遵守
2. `profile/_index.md` → `basics.md`、`preferences.md`、`links.md`
3. 按 JD 方向挑出相关经历，读对应 `profile/experiences/*.md`
4. 按目标企业性质读 `methodology/风格库.md` 对应节；按岗位读 `methodology/岗位速查.md` 对应节（没有对应节就现场研究该岗位话语体系，并把结论沉淀成新的一节）

注意 `preferences.md` 中带时效的信息（到岗时间等），过期就先问用户更新。

**空库守卫**：如果 `profile/_index.md` 的经历表只有占位行，且 `profile/experiences/` 下除 `_TEMPLATE.md` 外没有经历文件，立即停下——先带用户走 `workflows/intake.md` 建库（没有简历也能走「零起点建库」访谈）。禁止拿 `template/resume.html` 的示例内容投递。

## 2. JD 分析（展示给用户）

按方法论的 JD 分析框架输出：硬性门槛比对（逐条 ✓/✗）、高频关键词、加分项、风格信号、**命名执行单**。

**命名执行单（必须在生成前确定）**：
- `JD 原文`：逐字摘录命名要求；JD 没有要求时写「未指定」
- `命名模板`：保留 JD 指定的字段顺序和连接符（如 `姓名_学校_岗位.pdf`）；没有要求时用 `学校-姓名-专业-年级.pdf`
- `最终文件名`：立即用 `profile/basics.md`、`profile/preferences.md` 和本次 JD 中的真实值替换模板字段，写出包含 `.pdf` 的实际文件名（如 `李华_XX大学_量化研究实习生.pdf`），后续步骤统一引用这个值
- 模板字段缺值、简称口径不明，或 JD 同时出现互相冲突的命名要求时，必须先问用户；禁止保留 `{姓名}`、`XX学校` 等占位符，也禁止擅自猜测
- JD 指定的空格、`-`、`_`、`+` 等连接符原样保留；只有操作系统不允许的字符才需要调整，调整前先告知用户
- 第一次遇到某公司的命名要求时，记入 `methodology/岗位速查.md` 对应公司章节（方便下次投递同公司其他岗位时直接调用）

从这里开始，将命名执行单中的 `最终文件名` 视为不可丢失的交付约束。只识别出命名规则但没有得到实际文件名，不得进入第 4 步。

硬性门槛有 ✗ 时明确指出风险，让用户决定是否继续。

## 3. 经历选配（和用户确认后再动笔）

给出方案并等用户确认：

- 选哪几段经历、排什么顺序、每段主打什么角度（引用经历文件里的「现成表述」角度）
- 舍弃/压缩哪些、为什么
- 若关键经历「完整度：低/中」且 JD 高度相关：现场按方法论「深挖六问」问 3-5 个事实问题（答案回写经历文件），或建议先走深挖工作流
- 涉及【待确认】标记的事实（如证书表述、专业口径）第一次用到时必须问清，确认后更新 profile
- 同一次确认中加入“页眉视觉选择”，让用户分别选择：
  - 证件照：使用 `profile/basics.md` 已确认文件 / 重新上传 / 不使用。没有已确认文件时只能选择上传或不使用
  - 学校校徽：使用已确认文件 / 重新上传 / 帮我搜索 / 不使用。只有用户明确选择“帮我搜索”才做 Web Search；展示候选来源并等用户确认后再使用
  - 用户以前选择过也应简短展示当前选择，允许本次投递更改；不得默认补图

**交互方式**：
- 支持 `AskUserQuestion` 的 agent（如 Claude Code）：用结构化选项让用户点选
- 不支持的 agent：列出选项编号，让用户回复编号或文字

用户确认后才进入生成。用户如说「直接出」，跳过讨论但【待确认】事实仍必须问。

## 4. 生成简历

1. 建目录 `applications/YYYY-MM-DD-公司-岗位/`（当天日期）
2. 立即创建 `jd.md`，保存 JD 原文、分析和第 2 步的完整命名执行单；后续不得重新推导另一套文件名
3. 用户未指定版式时复制 `template/resume.html` 进去，只改内容和实际需要的区块，不改固定版式骨架；用户明确指定其他模板时才切换。固定版式以 `methodology/默认简历版式规范.md` 为准；校徽和证件照按第 3 步的用户选择决定，不属于必须保留的版式元素
4. 改写规则：方法论三步（排雷 → 价值翻译 → 量化），关键词对齐 JD，风格对齐企业性质
5. **保真**：每个数字都能在经历文件事实清单里找到；缺的问用户，没确认就不写
6. JD 有硬性时间要求时，在技能区下加一行「补充信息」（到岗时间、每周天数，从 preferences.md 取）
7. links.md 中「适用场景」匹配的链接写进简历对应位置

## 5. 渲染与一页校验

- 先从第 2 步的 `最终文件名` 去掉末尾 `.pdf`，得到唯一的 `输出基名`；不要临时重算或改写
- Windows：`pwsh scripts/render.ps1 -Html "applications/<目录>/resume.html" -Name "<输出基名>"`
- macOS / Linux：`bash scripts/render.sh "applications/<目录>/resume.html" "<输出基名>"`
- `applications/` 下渲染时必须显式传入输出基名；脚本会拒绝省略参数，避免静默生成 `resume.pdf`
- 按 `workflows/export.md` 使用同一 HTML 生成同基名 DOCX，并把 DOCX 渲染成逐页图片确认只有一页、版式无缺陷
- 对用户确认使用的证件照和校徽先做标准化转换；确保 HTML、DOCX、PDF、PNG 中的图片框尺寸与位置一致

**文件名规则**（优先级从高到低）：
1. **JD 明确要求的格式优先**（如「姓名-学校-专业」「学校+姓名+应聘岗位」等）—— 从第 2 步提取的命名要求
2. 没有要求时，默认 `学校-姓名-专业-年级`（从 preferences/basics 取）
3. JD 指定的连接符原样保留；文件名不得包含路径分隔符或操作系统禁用字符

- 超页 → 按方法论删减顺序删内容重渲，直到通过。禁止把字号降到 10pt 以下
- 渲染通过后用 Read 看一眼 PNG，确认排版没有溢出、错行

## 6. 生成配套材料（同目录）

- `email.md`：按方法论邮件规范写好 **主题** 和 **正文**（直接可复制），末尾列附件清单
- `notes.md`：用了哪些经历/角度、关键词覆盖情况、本次讨论中的遗留问题，并记录最终文件名

## 7. 记账与沉淀（不能跳过）

1. 在 `applications/_log.md` 追加一行（日期/公司/岗位/渠道/状态=已准备）
2. **回写**：本次对话中用户确认的新事实 → 写进对应经历文件的事实清单，消掉已回答的「待挖掘」；涉及完整度变化就更新 `_index.md`
3. 全新经历 → 提示用户之后走沉淀工作流补录

## 8. 汇报

汇报前逐项检查：

1. `applications/<目录>/<最终文件名>` 确实存在，且 PDF 文件名与 `jd.md` 命名执行单逐字一致
2. 同名 PNG 确实存在（仅扩展名改为 `.png`）
3. 同名 DOCX 确实存在（仅扩展名改为 `.docx`），且最新 DOCX 渲染只有一页并已目视通过
4. PDF 恰好一页，PNG 是从该 PDF 生成的一张 A4 图片；三种格式内容一致
5. `email.md` 附件清单引用的文件名与最终文件名逐字一致
6. 最终消息中的附件名来自磁盘实际文件，不凭记忆填写

任一项不一致，先修正或重新渲染，不得汇报「已完成」。全部通过后，在最终消息里给出：产物文件清单（可点击路径）、3 个关键定制点、复制粘贴指引（邮件主题贴哪、正文贴哪、附件是哪个文件）。
