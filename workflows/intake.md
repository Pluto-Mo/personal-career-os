# 沉淀工作流（intake）

> 触发：用户提到新实习/比赛/证书/链接，或发来简历、成绩单等材料。
> 目标：任何关于「我」的新信息，进来一次就永久沉淀，之后所有投递自动可用。

## 1. 拿材料

- 用户给了文件路径（docx/pdf/md/txt）→ 读取并提取文本（docx 可用 Python zipfile 解 `word/document.xml`）
- 是口述文字 → 直接用
- 都没有 → 问用户：「这次要沉淀什么？发文件或直接说都行。」

## 2. 与现有 profile 做 diff

读 `profile/_index.md` 和相关文件，把新材料内容分三类：

- **已有**：跳过
- **冲突**：新旧说法不一致（数字、头衔、时间）→ 列出来问用户以哪个为准，答案落库并注明来源
- **新增**：新经历或新事实

## 3. 补缺口（一轮 3-5 问）

对新增内容按方法论「深挖六问」问最关键的 3-5 个问题；如果是首次建库或材料很薄，用 `methodology/挖掘问题库.md` B 节的六轮普查清单。只问事实（做了什么、多少、结果），不问动机和感受。用户不想答就先落库、缺口记入「待挖掘」。

## 4. 落库

- 新经历 → 在 `profile/experiences/` 建文件，格式对齐现有文件：frontmatter（类型/组织/角色/时间/标签/适配方向/完整度）+ 一句话 + 事实清单 + 现成表述 + 待挖掘
- 已有经历的新事实 → 追加进事实清单，标注来源；能升级「现成表述」就顺手打磨
- 基础信息类（证书、技能、链接）→ 更新 `basics.md` / `links.md` / `preferences.md`
- 链接类信息一律登记进 `links.md` 并写清「适用场景」

## 5. 更新索引

`profile/_index.md`：新经历加行、完整度调整、全局待办增删。

## 6. 汇报

新增/更新了哪些文件，还剩哪些「待挖掘」值得之后走深挖工作流。
